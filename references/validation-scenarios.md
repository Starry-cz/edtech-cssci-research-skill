# 验证场景矩阵

更新 Skill 后，用下表的代表性任务检查路由是否正确。验证关注“是否给出可执行、可核验的研究支持”，不以生成看似完整的论文段落作为通过标准。

| 触发任务 | 预期资源或模式 | 核心输出 | 禁止行为 |
| --- | --- | --- | --- |
| “AI 教育很热，帮我定题” | `topic_diagnosis`；问题—模型—证据文件 | 对象、情境、机制、证据需求与候选问题 | 把政策口号或技术名称当作研究问题 |
| “围绕 AI 教育写一篇论文，主线怎么搭” | `argumentation_blueprint`；论证与修订工作底稿 | 问题—回答—证据—贡献—边界主张图与待补材料 | 用修辞性创新点或泛化效果填补论证空白 |
| “把这个热点改成研究问题” | `research_question_refinement` | 可回答问题、概念边界、证据与不可回答部分 | 用宽泛变量关系替代研究问题 |
| “帮我列论文提纲” | `outline_building`；章节收束文件 | 章节功能、证据接口与论证顺序 | 只按常见章节名机械罗列 |
| “我的全文主线和贡献不清楚” | `argumentation_blueprint`；论证与修订工作底稿 | 问题—回答—证据—贡献—边界主张图 | 以修辞性创新点替代可支持的贡献 |
| “模型中的变量为何这样关联” | `conceptual_model_building`、`framework_defense` | 构念定义、机制、替代解释和检验路径 | 把要素清单画成理论模型 |
| “设计一个智慧课堂/平台模型” | `practice_model_design`；模型与系统设计文件 | 设计原则、活动链、实施条件和过程/结果证据 | 把功能或上线写成教育效果 |
| “文献很多，综述写不出来” | 两个文献模式；综述矩阵 | 检索路径、主题矩阵、争议和缺口 | 编造出处或作者罗列 |
| “核对这条引文能否支撑主张” | `citation_and_evidence_check` | 主张—来源映射与核验状态 | 未读原文即确认其结论 |
| “有问卷、课堂或平台数据，怎么分析” | `education_data_analysis`；分析报告模板 | 数据审计、分析计划、诊断、稳健性与边界 | 跳过分析单位、时间、缺失和隐私检查 |
| “我想预测并用 SHAP 解释” | `interpretable_ml_analysis`；可解释机器学习文件 | 泄漏控制、切分、模型比较、解释与稳定性 | 将预测或特征重要性写成因果机制 |
| “层次分类重跑后如何报告” | 可解释机器学习模式；层次分类文件 | 层级标签规则、切分、误差追踪与可视化 | 在层级间复用泄漏信息或只报最好指标 |
| “帮我写摘要/引言/讨论” | `section_revision`；写作工作流 | 章节承担的判断、所需证据、修复方案和边界 | 将摘要承诺、结果、讨论和贡献混写 |
| “全文看起来像材料拼贴” | `draft_review`、`argumentation_blueprint` | 风险分级、重构优先级和证据缺口 | 先润色语句而不处理主线断裂 |
| “实证结果、讨论和贡献总是混在一起” | `draft_review`、`section_revision`、论证文件 | 三者分工、对应证据和结论降级方案 | 用显著性直接宣布理论或实践贡献 |
| “模型/实践论文怎样证明有效” | `practice_model_design`、`draft_review` | 设计理由、实施过程、证据类型与可支持结论 | 用系统演示、案例展示代替效果证据 |
| “政策比较或人机协同案例怎么写” | 政策比较与人机协同文件 | 比较维度、语料边界、编码规则和替代解释 | 将政策背景或个案叙述当作普遍效果 |
| “收到审稿意见，帮我回复” | `reviewer_response_and_revision` | 意见台账、回应、修改位置、证据与状态 | 只写礼貌套话或承诺无法验证的修改 |
| “改了研究问题，会影响哪里” | `revision_cascade`；论证与修订工作底稿 | 修改层级、摘要/图表/附录/回应函传播清单和验证 | 只改正文一处而忽略连锁更新 |
| “投稿前查一下” | `reference_and_artifact_audit`、`pre_submission_check`、期刊核验文件 | 阻断项、工件清单、官方核验与提交判断 | 编造 CSSCI 目录、期刊格式或栏目要求 |
| “帮我记录项目决策/降低模板腔” | 项目记忆与表达风险文件 | 决策留痕、具体化改写与事实核验提示 | 以规避检测为目标或伪造个人经历 |
| “给 Claude、ChatGPT、Gemini 或 Trae 用” | `platforms/` 与通用核心指令 | 适配安装/粘贴方式与同一证据边界 | 声称所有平台可自动读取本仓库 |

## 统一通过条件

- 输出能明确本次任务的主模式、材料状态、核心判断、证据来源和结论边界。
- 文献、数据、期刊要求和审稿意见均只基于已给材料或已核验来源；未知项保留为待核验。
- 数据分析包含过程审计、验证或稳健性判断；模型/实践研究包含设计理由和相称的证据类型。
- 写作或返修任务先处理主张、理论、设计和证据，再处理段落与措辞；一项结构性修改会检查摘要、图表、附录、回应函等连锁位置。
