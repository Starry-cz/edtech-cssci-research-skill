# 审稿回应与修订治理

用于导师意见、同行评阅、匿名审稿和多轮返修。目标是让每条意见、修改动作、证据变化和未解决分歧可追溯，而不是只生成一封礼貌回应函。

## 一、先建立修订输入

开始前识别：当前稿件版本、目标期刊、意见来源、审稿轮次、截止时间、作者已作出的关键决定，以及用户本轮是否授权直接修改文件。只有“评审/诊断”请求时，不擅自改稿；用户明确要求修改时，按确认的范围实施。

把意见拆成原子任务。一个意见若同时涉及理论、方法和表达，应拆成多个编号，避免以一次局部改写假装全部回应。

对每个原子任务补记“首要修复层”：研究问题、综述缺口、理论与构念、研究设计与证据、结果与结论、章节结构，或表达与格式。先处理会改变其他章节的层级，后处理句段和格式；不能先润色后发现主问题、样本或结论已改变。

## 二、意见分类与优先级

按问题性质分类：

- **事实与来源**：文献、数据、政策、术语、时间、机构或图表信息有误。
- **论证与框架**：问题未闭合、维度依据不足、替代解释缺失、结论越界。
- **研究设计与分析**：抽样、测量、识别、数据泄漏、模型验证或复现不足。
- **扩展请求**：增加案例、变量、理论、稳健性检验或文献。
- **呈现与格式**：结构、图表、匿名、引文、语言和期刊格式。
- **偏好与冲突**：属于写作偏好，或与其他审稿意见、数据条件及论文边界冲突。

严重度统一映射为：

- `submission blocker`：真实性、伦理、核心设计、证据链或期刊硬性要求问题。
- `major revision`：会影响研究问题、理论解释、方法可信度或主要结论的问题。
- `polish`：不改变实质判断的表达、顺序和格式问题。

不得用固定数量、工作时长或“审稿人说了就必须做”代替判断。

## 三、建立意见—行动矩阵

每条意见至少记录：

| 字段 | 内容 |
|---|---|
| ID | 审稿人/轮次/序号，如 `R2-C03` |
| 原始意见 | 保留准确语义；长意见可概括并附原文位置 |
| 问题性质与严重度 | 分类 + `submission blocker / major revision / polish` |
| 证据与判断 | 草稿位置、数据或文献依据，以及是否同意该意见 |
| 处理决定 | `accept / partial / decline / needs verification` |
| 修改动作 | 增加、删除、移动、重算、补证据、缩窄、降级措辞或解释不修改 |
| 修改位置 | 章节、页码、段落、图表或文件；定稿后更新准确定位 |
| 首要修复层与连锁项 | 先修改什么，以及题目、摘要、理论、数据、图表、附录、结论和回应函中哪些必须同步检查 |
| 验证方法 | 如何确认已解决，如重跑分析、交叉核引、匿名扫描 |
| 状态 | `open / implemented / verified / deferred` |

`partial` 或 `decline` 必须说明学术依据、材料限制和替代处理，不用防御性语气。若意见指出事实错误，先核验再改；不能因其来自审稿人而直接采用未经证实的信息。

## 四、多位审稿人的合并与冲突处理

先把意见整理为：

1. **共识项**：多位审稿人指向同一缺口，通常优先处理。
2. **互补项**：分别涉及问题、理论、方法或呈现，可组合成一项结构性修改。
3. **冲突项**：例如一位要求扩展理论，另一位要求压缩篇幅。
4. **独有项**：只出现一次，但可能仍是阻断问题。

冲突项不得静默择一。写明两种处理的收益、代价、对研究问题和篇幅的影响，再选择能最大程度保留证据链与期刊适配性的方案。需要作者价值判断或会改变研究范围时，提交决策而不是代替作者决定。

## 五、回应函写法

每条回应采用四步：

1. 准确复述对方关切，不夸大已完成工作。
2. 给出处理决定与理由。
3. 概述实质修改或解释为何不修改。
4. 标明修改后的准确位置，并在必要时摘录少量关键新文本。

避免“已认真修改”“已全面补充”等不可验证套话。方法或结果发生变化时，应同步说明数据版本、分析代码、图表、摘要和结论是否更新。

## 六、修订台账与版本检查点

每轮结束生成：

- 已完成项及验证证据。
- 尚未解决项、原因和责任人。
- 结构、理论、数据、分析与结论发生的连锁变化。
- 下一轮需要核验的期刊规则、文献或数据。
- 当前稿件、数据、代码、图表和回应函版本标识。

涉及重算、删样本、改变量、换理论框架或改变主要结论时保存检查点。不得只改正文而不更新摘要、图表、附录、参考文献与回应函。

若重跑或审稿意见暴露测试集选模、泄漏、准则污染或不可复核结果，回应函应如实说明修复动作与验证计划，不得把受影响的性能或因果结论宣称为已经确认。修复后同步更新相应表图、方法和结论；这些审计说明不自动写入正文的摘要、讨论或结论。

若意见要求“加强贡献”或“强化实践意义”，先判断证据最多支持提出、描述、关联、预测、检验还是设计原则。必要时补证据、缩窄结论或把建议改为条件化行动，不能用更强的动词掩盖证据不足。

## 七、教育技术学专项联动检查

- 新增技术功能描述后，是否补充实际使用、教学活动和过程证据。
- 变量、平台日志或算法定义变化后，理论构念与结果解释是否同步。
- 模型性能或 SHAP/ALE 结果重跑后，图表、排序和结论边界是否一致。
- 干预或设计原则调整后，实施忠实度、迭代证据和适用情境是否更新。
- 审稿人要求“强化实践意义”时，建议是否仍由具体发现导出，而非改成宣传口号。
