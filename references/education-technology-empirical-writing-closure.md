# 教育技术学实证论文：从分析链到论证闭环

用于学习分析、教育数据挖掘、课堂观察、问卷与混合实证研究的写作、改写与全文诊断。目标是把“已有分析”组织成教育技术学论文的中心回答，而不是模仿某篇论文的题目、句式、结论或图表。

本文件从已提供的可解释机器学习研究和教育技术学期刊公开摘要中提炼可迁移规则：教育问题居前，理论和构念组织证据，方法服务于回答，结果与讨论各司其职。它不承诺期刊录用，也不替代目标期刊的当期规范。

## 一、先把研究材料闭合为一条论证链

实证论文至少应能回答下列映射：

```text
教育问题 → 理论/分析框架 → 构念或过程维度 → 可观察指标与材料
→ 证据序列 → 教育学解释 → 相称的使用或改进含义
```

这不是要求每篇论文都画一张复杂框架图。它要求题目、摘要、研究问题、变量分组、结果小节、讨论和建议使用同一组核心构念；如果后文出现的维度不能回溯到前文的理论或分析框架，就应补足依据、重组为探索性发现，或从主线中删除。

写作前可填写 `assets/empirical-writing-closure-canvas.md`。该画布只用于内部构思，字段名不得进入投稿正文。

## 二、方法在论文中的位置：证据阶梯，而非章节主角

机器学习或复杂统计不是论文的中心回答。它们分别承担不同证据任务；只有实际完成并报告的步骤，才可进入下列阶梯：

| 证据层 | 要回答的问题 | 正文中的合适位置 |
| --- | --- | --- |
| 比较或基线 | 为什么这一分析方案足以进入后续解释 | 方法末或结果开端，用少量文字交代 |
| 构念/维度差异 | 哪些教育过程信息更能区分或关联结果 | 结果主体，按构念归并表图 |
| 关系形态或情境差异 | 这种关系在不同区间、群体或情境中怎样变化 | 结果后段，说明图表所见事实 |
| 稳定性、反例或使用条件 | 哪些发现更值得进入中心判断，适用到哪里 | 结果收束、讨论或附录，按实际证据决定 |

不要把阶梯机械写成“四个小节”或“四项创新”。没有模型比较时不补造“最优模型”；没有异质性、时间或过程材料时，不把并列特征改写为已验证的生成链或干预机制。

## 三、摘要：先完成教育回答，再压缩方法

教育技术学的非结构式摘要通常不是“结果表的压缩版”，而是一次完整但紧凑的研究回答：

1. **问题入口**：指出特定教育活动中的矛盾、解释缺口或实践难题。
2. **研究行动**：用一句交代对象/材料、分析框架和方法功能。
3. **发现簇**：选择最能回答问题的二至四项发现；数量由研究问题决定，不强制凑足“四点”。
4. **综合判断与意义**：把变量或图形归并为过程结构、类型差异、设计认识或解释性综合，再落到相称的评价、观察、设计或改进含义。

对预测或可解释机器学习研究，优先从“变量列表”上升到“构念/过程维度”。例如，多个提问、应答、思维或反馈指标可以在理论支持下被概括为课堂认知参与与教学调节的结构；但若缺少时序、交互或干预证据，不把这种结构写成某个教学行为必然导致结果的因果链。

摘要中通常只保留使读者理解研究类型所必需的数据规模、材料类型、方法功能和至多一个关键性能锚点。模型清单、解释器名称、参数、所有重要性比例、曲线细节和复现检查进入方法、结果表图或附录。具体信息带宽规则见 `abstract-state-and-evidence-control.md`。

## 四、各章节如何共同回答问题

| 章节 | 本节应完成的判断 | 写作时优先检查 |
| --- | --- | --- |
| 引言 | 为什么这个教育现象值得解释，已有研究留下了什么可回答的缺口 | 从教育活动和研究矛盾进入；政策或技术趋势只作情境，不能代替问题 |
| 综述/理论 | 采用何种构念及其关系来理解问题，为什么这些维度进入分析 | 形成“理论—构念—指标/编码—预期解释”的映射，而非作者罗列 |
| 方法 | 材料、分析单位与方法为何能产生回答问题所需的证据 | 逐项说明方法的研究功能，不写算法教材或软件日志 |
| 结果 | 证据对问题说明了什么 | 按研究问题和发现簇呈现，先描述表图事实，再给最小经验判断 |
| 讨论 | 多项发现合起来改变了什么理解 | 与理论和文献对话，形成过程结构、类型解释或条件化认识；不重复表图 |
| 结论/启示 | 对题目作出最小但明确的回答，并说明可怎样使用 | 每项建议能回溯到具体发现、对象与场景；不把建议写成效果保证 |

结果和讨论的分工尤其重要：结果说明“发现了什么”；讨论才回答“这些发现合起来意味着什么、与既有解释有何关系、可在何种情境中使用”。同一段既连续报数又宣称理论贡献时，先拆开，再决定哪一部分确实需要保留。

## 五、把“解释性综合”写得有教育学意义

解释性综合不是口号，也不等于因果断言。它应同时满足：

- **可回溯**：综合中的每个过程维度都能回到变量、编码、图表或材料；
- **有依据**：理论或已有研究解释为何这些维度可以放在同一分析框架中；
- **有区分**：说明它相较单一变量排序、传统线性解释或既有分类新增了什么认识；
- **有条件**：建议明确面对的主体、活动或使用情境，不把样本内关联写成普遍处方。

可使用“呈现出……过程结构”“显示……之间的关联配置”“为理解……提供分析视角”等表达；只有设计提供了相称的时序、过程或干预证据，才使用“导致”“提升”“形成机制已被验证”等强因果措辞。

## 六、全文诊断的四个高频失配

| 失配 | 识别信号 | 优先修复 |
| --- | --- | --- |
| 问题与框架脱节 | 引言提出一个问题，变量、结果或结论却回答另一个问题 | 先重写中心问题和构念映射，再处理章节语言 |
| 框架与证据脱节 | 框架图出现的维度未测量，结果变量又未在理论中出现 | 核对理论、变量表、图表和讨论的同名构念 |
| 结果重复或工具化 | 多处重复模型性能，或按“模型—SHAP—ALE—稳健性”作为正文主线 | 合并基础性能说明，按发现簇重排结果 |
| 讨论与启示失配 | 讨论重报结果，建议只有口号或与证据无关 | 用“发现—解释—具体使用条件”重写，并删除无法回溯的建议 |

诊断优先提交上述结构问题，再处理措辞。不要为了显得“更像 C 刊”加入固定的四点发现、三条建议、长段局限或特定作者口吻。

## 七、生成前的最小检查

在生成摘要、结果、讨论或结论前，先确认：

1. 本文的中心教育问题和最终回答各是什么？
2. 理论构念怎样进入变量、编码或分析维度？
3. 每一项核心发现由哪张表、图或材料支持？
4. 讨论的综合判断相较结果新增了什么解释，而非重复了什么数字？
5. 建议的主体、场景和证据来源是什么？

任一事实缺失时，不在最终正文中写“待补充”“投稿时完善”等兜底语；如用户要求终稿，缺失事实应在稿外阻断清单中说明。
