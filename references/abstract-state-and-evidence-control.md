# 教育技术学摘要：从方法报告到学术判断

用于起草、改写或诊断中文教育技术学论文摘要。摘要服务于编辑和同行的第一次判断：研究在回答什么教育问题、依据什么材料、发现了什么、推进了什么认识。它不是方法附录，也不是风险声明书。

## 一、默认组织方式

非结构式摘要通常完成四次信息推进：

1. **问题**：从一个具体教育矛盾、解释缺口或实践困惑进入。
2. **研究行动**：用一句交代对象/材料规模、研究类型和方法功能。
3. **发现簇**：按教育构念、过程维度或比较类型呈现二至四项关键发现，不按分析代码顺序罗列。
4. **中心判断**：说明发现如何改变对教学、学习、评价或技术应用的理解，并给出相称的实践或研究意义。

研究类型、期刊字数和是否结构式摘要，以目标期刊当期官方要求为准；不复制特定作者或期刊的句式。

## 二、可解释机器学习摘要的社会科学写法

先完成“原始变量 → 教育构念 → 过程维度 → 中心判断”的压缩。例如，问题认知要求、学生应答、思维表达和反馈可以被组织为课堂认知参与与教学调节的过程结构，而不是逐项呈现特征排行榜。

- 方法通常只需说明“基于何种数据，采用何种建模/解释方法回答何种问题”。完整模型清单、超参数、SHAP/ALE、Bootstrap、消融和所有性能指标留在方法或结果表图。
- 发现优先写教育上的区别、关系形态或过程配置；性能数字只在它本身构成中心发现或改变读者理解时保留一个锚点。
- 允许使用“形成”“塑造”“机制”“共同”“何以达成”等研究问题和理论解释语言。不要因为出现这些词，就自动改写成“模型稳定依赖”“当前留出样本”之类的技术口吻。
- 当论文是预测研究时，模型结果可以支持有理论依据的教育学解释和评价启示；若用户未要求方法审计，不必在摘要追加“并不构成因果”或“仍需外部验证”的固定结尾。

## 三、摘要中的方法事实与方法审计分开

| 内容 | 放在摘要 | 放在方法/结果/审计说明 |
| --- | --- | --- |
| 数据类型、样本规模、研究对象 | 通常保留 | 补充抽样、清理和编码细节 |
| 建模方法的研究功能 | 通常保留一句 | 算法清单、参数、训练流程 |
| 核心发现及其理论归并 | 优先保留 | 全部变量排序、所有图表数值 |
| 验证与稳健性 | 仅在构成中心发现时简要保留 | 切分、调参、稳定性、消融和复现细节 |
| 方法问题 | 仅在它会改变摘要某项核心承诺时改写该句 | 用户要求的方法审计或投稿预审中单列 |

不要为了“严谨”把“同一测试集参与选模、需要嵌套交叉验证、外部样本仍待检验”等审计事项自动移入摘要。它们属于方法检查；只有作者正在主张独立最终性能、严格阈值或确定因果效应时，才需要针对这项承诺处理。

## 四、内部摘要卡

先填写 `assets/abstract-argument-card-template.md`，再删除其字段生成连续摘要。内部核对的目的，是确保摘要兑现研究，而不是要求把每个条件都呈现出来。

完成后检查：

- 删除算法和数字后，读者是否仍知道为什么研究、发现了什么、意味着什么；若不知道，重写中心判断。
- 是否把多项结果归并为教育上的发现簇，而非罗列“模型性能—重要性—曲线—稳定性”。
- 是否让最后一句落在具体的理论推进、课堂观察、评价优化或教学改进上，而不是“提供线索/仍需验证”的防御性套话。
- 仅在摘要的某一强承诺与研究设计直接冲突时，局部调整该承诺；不要用通用免责声明接管整段摘要。

## 五、供方法审计使用的核对项

只有用户要求检查方法、复现或投稿前技术质量时，才核对训练/验证/测试角色、预处理、目标—特征重叠、群组/时间切分、性能不确定性、SHAP相关特征和ALE稀疏区。审计输出单列“发现—影响—修复”，不自动生成论文局限段；具体规则见 `interpretable-machine-learning.md`。
