# 项目记忆与决策日志

用于持续数周或多轮迭代的论文项目。它保存“已经确认的事实与决定”，减少不同会话之间的漂移，但不能把待核验内容固化成事实。

## 一、何时建立

当任务出现下列任一情况时，建议创建项目上下文：多轮修订、多个作者、多个数据/代码版本、目标期刊切换、术语反复变化、审稿意见往返，或存在尚未解决的关键决策。

新项目可复制 `assets/project-context-template.md`。项目上下文应放在论文项目目录，不把作者隐私、未公开数据或本地绝对路径提交到公开 skill 仓库。

## 二、记录什么

- 论文问题、分析单位、场景、最小贡献和当前结论边界。
- 目标期刊及其规则的官方来源、核验日期和版本。
- 数据、代码、图表、稿件、附录与回应函版本对应关系。
- 理论、构念、变量、指标、编码和术语决定。
- 已拒绝的方案及理由，避免下一轮无意恢复。
- 已核验事实、待核验项、未解决分歧和下一步动作。
- 伦理审批、数据权限、匿名化与利益冲突状态，但不复制敏感凭证。

## 三、状态规则

每项信息标记为：

- `verified`：有可定位的原始材料或官方来源。
- `author decision`：作者明确选择，记录日期和理由。
- `working assumption`：为推进工作暂定，不能写成研究事实。
- `needs verification`：缺来源、字段或复核。
- `superseded`：已被新决定替代，同时保留变更原因。

事实、推断和作者偏好必须分栏。不要把“以前这样写过”当作继续保留的依据。

## 四、变更传播检查

关键决定更新后，检查其下游影响：

```text
研究问题/术语 → 理论与模型 → 变量或编码 → 分析 → 图表
              → 摘要与正文 → 结论边界 → 投稿材料与回应函
```

例如改变预测目标，不只更新方法，还要检查特征泄漏、性能指标、解释图和摘要结论；改变模型维度，应检查操作化、案例编码、图示和框架贡献。

## 五、维护节奏

只在出现新事实、新决定、版本切换或状态改变时更新，不把每次语言润色都写入。每轮开始先读取“当前版本、已确认决定、开放问题”，每轮结束更新“完成项、未完成项、下一检查点”。

若项目记录与当前文件冲突，以可核验的最新材料为准，并把旧记录标为 `superseded`，不得依靠模糊兜底继续沿用。
