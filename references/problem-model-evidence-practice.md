# 教育技术学“问题—模型—证据—实践”研究架构

用于概念模型、技术应用框架、教师发展模式、智能教育系统、课程改革、政策比较和实践研究。目标不是为模型起一个新名称，而是让问题、理论、结构、数据、活动、结果与边界能够互相追溯。

## 一、先识别难题性质

同一选题可包含多类难题，但必须指出主难题及其证据：

- **结构性难题**：规模化供给与个性化支持冲突，主体、资源或层级配置不匹配。
- **机制性难题**：理论知识难以转成教学行动，数据、资源、角色或反馈未形成有效连接。
- **效能性难题**：过程粗放、诊断滞后、反馈不准，技术虽被使用却未改善关键活动。
- **发展性难题**：静态评价与动态成长错配，职前职后、课内课外或不同阶段不能贯通。
- **价值性难题**：效率目标挤压育人、公平、自主、福祉或教师专业判断。
- **治理性难题**：责任、权限、隐私、安全、标准、监管和多主体协同不清。

“存在不足”“技术应用不深”不构成充分诊断。至少交代谁遇到什么阻碍、发生在哪个活动、现有做法为何失效、可观察到什么后果。

## 二、选择与问题相称的论文架构

### 1. 需求—愿景—路径

适合宏观战略和系统变革研究。需求说明教育矛盾，愿景按学习者、教师、学校、家庭、治理或评价等主体与场景展开，路径逐项回应需求。不得以政策口号代替需求证据。

### 2. 历史—机制—比较—路向

适合技术演进与国际态势研究。历史说明变化而非罗列年份；机制解释为何产生质变；比较必须使用共同维度；本土路向需结合制度、资源和对象差异。

### 3. 概念—风险/挑战—策略—案例

适合新概念、生成式人工智能和人才培养研究。先限定概念的组成与边界，再把挑战映射到策略；案例展示策略如何运行，不能凭单一案例宣称普遍效果。

### 4. 指标—系统—场景—反馈

适合画像、智能测评、推荐和实训研究。先把能力或状态拆成可观测指标，再说明数据如何进入系统、系统如何在活动中反馈、反馈如何支持行动，并报告误差、权限和伦理边界。

### 5. 问题分析—理论建构—模式设计—迭代应用

适合设计型和长期实践研究。每轮迭代报告问题、修改、证据和理由；模式图中的环节须在系统功能、教学活动或研究材料中有对应物。

### 6. 比较框架—文本编码—共性/差异—本土转译

适合国际政策研究。框架可来自理论，也可由预分析形成；说明语料、时间、编码、复核与饱和/一致性。共性不是简单求同，差异应联系政策目标和治理条件。

## 三、把理论变成模型部件

先建立“理论—部件—证据”映射：

| 理论功能 | 模型中的表现 | 需要的证据 |
|---|---|---|
| 定义研究对象 | 维度、层级、角色或状态 | 概念文献、政策标准、专家意见 |
| 解释作用过程 | 箭头、循环、中介、调节或反馈 | 过程数据、实验、访谈、日志或理论命题 |
| 约束设计 | 任务、支架、角色授权、反馈规则 | 设计说明、实施记录、形成性证据 |
| 指导评价 | 指标、观测点、时间窗口、比较基准 | 测量证据、编码本、信效度或一致性 |
| 界定边界 | 失效条件、风险、责任和情境 | 反例、异质性、敏感性与伦理分析 |

若某个理论只在综述中出现、在设计和解释中消失，标记 `Theory Decoration`。若模型箭头没有理论命题或过程证据，标记 `Mechanism Gap`。

## 四、构建可追溯的五层模型

不要求所有研究都画五层，但必须回答五类问题：

1. **目标层**：改善哪项学习、教学、教师发展或治理问题。
2. **构念层**：核心能力、状态、关系或价值如何定义。
3. **数据层**：从何种场景采集哪些数据，数据粒度、质量、权限和时间如何控制。
4. **分析与交互层**：使用何种规则、算法、智能体或协作机制，如何产生解释、推荐、诊断或反馈。
5. **服务与评价层**：谁根据输出采取什么行动，如何评价过程、结果、风险和持续改进。

画像和推荐研究还要区分：直接观测数据、推断标签与预测标签；核心场景数据、跨场景拓展数据与治理环境数据；个人画像与群体画像。标签数量不是贡献，标签与教育构念、观测证据及行动建议的对应关系才是。

## 五、建立证据闭环

按研究问题选择证据，不用一种证据替代全部判断：

```text
问题证据 → 构念/理论证据 → 设计证据 → 实施证据
        → 过程证据 → 结果证据 → 解释与边界
```

- 问题证据说明难题真实存在，不能只靠政策导语。
- 构念证据支持维度和关系，不能只靠研究者命名。
- 设计证据解释为什么采用特定任务、技术和角色。
- 实施证据说明方案实际如何运行以及是否忠实执行。
- 过程证据连接技术使用与行为、互动、认知或情感变化。
- 结果证据回答研究问题，不能把展示、好评或使用次数当作效果。
- 边界说明样本、场景、时间、误差、伦理和替代解释。

模型构建、系统开发和应用验证可在一篇文章中出现，但要区分“提出”“实现”“可用”“初步有效”“机制成立”和“可推广”六种不同强度的结论。

## 六、让图表承担论证任务

每张图表只承担一个主任务，并在正文中解释读图逻辑：

- **研究路线图**：问题、阶段、方法、产出与验证关系。
- **理论/机制图**：构念、方向、反馈、层级和边界条件。
- **系统架构图**：数据输入、处理、模型、交互、输出和使用主体。
- **结果结构图**：类型、层级、路径、群体或时间差异。
- **应用流程图**：用户行动、系统反馈、教学决策和下一轮改进。
- **指标表**：维度—指标—观测点—数据来源—算法/编码—解释边界。

自检每个节点：是否在正文定义，是否有材料支持，是否在方法中实现，是否在结果中被使用。装饰性箭头、无法验证的“闭环”和把多个不同逻辑塞入一图都应重画。

## 七、从结果导出分层策略

建议不得从常识直接跳出。使用“发现—对象—行动—条件—指标”格式：

```text
基于[具体发现]，由[责任主体]在[场景]采取[行动]，
依赖[资源/规则/能力条件]，并用[过程或结果指标]跟踪。
```

若研究识别了不同类型或层级，应提供差异化干预；若只验证总体相关关系，不应编造精细人群画像。政策比较得出的建议要说明哪些经验可转移、哪些受制度差异限制。

## 八、人机协同与伴随式评价

教育人工智能研究至少说明：

- 人保留哪些目标设定、解释、审核、申诉和最终决策权。
- 系统在何时采集什么数据，输出如何进入教学或学习活动。
- 自动反馈如何经教师或学习者判断后转成行动。
- 模型更新、错误纠正、版本变化和长期追踪如何记录。
- 效率、公平、自主、福祉、隐私和专业判断发生冲突时如何处理。

“人在回路”不是一句原则，应落实为可见的审核节点、权限配置、反馈修订、异常处置和责任记录。

## 九、投稿前专项检查

- 主难题是否被证据化，后文是否持续回应。
- 论文架构是否与研究类型相符，是否把概念文章伪装成实证研究。
- 理论是否进入模型部件、设计决策和结果解释。
- 模型命名之外是否存在可检验关系、可操作流程或可迁移原则。
- 图表节点是否与正文、数据、方法和结果逐项对应。
- 案例、调查、相关、聚类、预测和短期前后测是否被准确表述。
- 应用策略是否由发现导出，并明确责任主体、条件和评价指标。
- 是否讨论技术失效、非预期后果、数据与算法限制及人的最终责任。

若矩阵、类型学、连续谱或多层模型是主要贡献，进一步读取 `framework-defense-and-figure-audit.md`，检验维度选择、竞争解释、层级一致性、边界案例、操作化和图文对应；不能因图形闭合就认定理论闭合。
